# VOTO Product Messaging Guide

## Core positioning

- VOTO is the regulatory intelligence layer for crypto.
- Use "VOTO - Verified Oversight & Trust Oracle" sparingly in high-context brand surfaces such as metadata, investor materials, or top-level product introductions.
- The product helps teams decide whether a crypto entity appears legitimate, regulated, risky, or connected to warning and sanctions signals.
- The graph is an investigation layer, not the main value proposition.
- The API is the monetization layer that embeds legitimacy checks into production workflows.

## Primary audiences

- Exchanges running pre-listing due diligence.
- Compliance teams at crypto and fintech firms.
- Funds and serious investors screening projects and issuers.
- Wallets, apps, and infrastructure providers embedding legitimacy checks through API.
- Enterprise buyers evaluating compliance intelligence tooling.

## Approved value statements

- Know if a crypto project is legitimate, instantly.
- VOTO - Verified Oversight & Trust Oracle
- Compliance intelligence for crypto platforms.
- Real-time legitimacy and regulatory checks for crypto entities.
- Use live regulatory, sanctions, and entity intelligence through a visual interface or API.

## What the product does

- Search an entity, domain, contract, wallet, or jurisdiction.
- Surface regulatory status, warning signals, sanctions exposure, evidence, and connected relationships.
- Keep source-backed facts separate from inferred links.
- Provide timeline visibility and monitoring for ongoing diligence.

## How to talk about the graph

- Use phrases like "investigate connected evidence", "follow linked relationships", and "see hidden relationships instantly".
- Do not describe the product as a generic graph platform.
- Do not lead with topology, ontology, clusters, or graph engine language in customer-facing surfaces.
- Customer-facing context docks should show live trust context, actions, or coverage state rather than generic workflow instructions.

## Evidence and trust language

- Prefer "source-backed", "evidence-first", "official-source", and "confidence-scored relationships".
- State clearly that official regulator and sanctions sources take priority over enrichment layers.
- Never imply legal conclusions the data cannot support.
- If coverage is limited, say "limited source coverage" or "source unavailable".

## Terms to prefer

- legitimacy check
- regulatory status
- sanctions exposure
- warning signals
- linked evidence
- due diligence
- compliance intelligence
- timeline of changes
- source-backed facts
- confidence-scored relationships

## Terms to avoid leading with

- graph engine
- topology
- ontology
- cluster map
- AI assistant
- analytics dashboard
- graph-first product

## CTA guidance

- Check a project
- Try live lookup
- Get API access
- Request API access
- Explore live intelligence
- Monitor entity

## Guardrails

- No fabricated entities, warnings, sanctions, or sample records in production surfaces.
- Example schemas in docs and API landing pages must be clearly marked as example schema only.
- The product should make sense to a compliance lead in under five seconds.
